01

핵심 철학

기법보다 먼저 깔려 있는 태도. 모든 결정의 바탕입니다.

취향은 타고나는 게 아니라 훈련된다

언제무엇을 만들지 막막할 때, ‘되게’가 아니라 ‘느낌이 맞게’ 만들고 싶을 때.
어떻게좋은 작업을 곁에 두고, 왜 좋은지 분해해 보고, 애니메이션을 리버스 엔지니어링하며 본다. 호기심이 곧 훈련.

보이지 않는 디테일이 쌓인다

언제‘아무도 눈치 못 챌 텐데’라며 디테일을 건너뛰고 싶을 때.
어떻게기능이 ‘당연히 이래야지’대로 동작하면 사용자는 멈칫하지 않는다. 개별로는 안 보여도 합쳐지면 보인다 — 수천 개의 작은 목소리가 화음을 이루듯.

아름다움은 레버리지다

언제기능은 충분한데 차별점이 없을 때.
어떻게사람은 기능이 아니라 전체 경험으로 도구를 고른다. 좋은 기본값과 좋은 모션은 진짜 차별점 — 소프트웨어에서 가장 저평가된 무기.
02

애니메이션 의사결정 프레임워크

코드를 짜기 전, 이 순서대로 답하세요: ① 애니메이션할까? → ② 목적은? → ③ easing은? → ④ 속도는?

① 이걸 애니메이션해야 하나?

언제새 인터랙션을 만들 때 가장 먼저. ‘얼마나 자주 보게 되나?’를 묻습니다.
어떻게자주 볼수록 모션을 줄이거나 없앤다. 키보드로 시작되는 동작(단축키·커맨드 팔레트)은 절대 애니메이션하지 않는다 — Raycast가 열고 닫을 때 애니메이션이 없는 이유.
빈도결정
하루 100회+ (단축키, 팔레트)절대 금지
하루 수십 회 (호버, 목록 이동)제거하거나 대폭 축소
가끔 (모달, 드로어, 토스트)표준 애니메이션
드묾·처음 (온보딩, 축하)딜라이트 추가 OK

② 목적이 뭔가?

언제‘왜 애니메이션하지?’에 한 문장으로 답할 수 없다면 멈추세요.
어떻게유효한 목적: 공간적 일관성(토스트가 같은 방향에서 들고 난다) · 상태 표시 · 설명 · 피드백(누르면 줄어듦) · 급격한 변화 방지. ‘멋있어서’인데 자주 본다면 → 하지 않는다.

③ 어떤 easing?

실제 예시
언제요소가 들어오거나(enter) 나갈 때(exit) → ease-out. 화면 안 이동/모핑 → ease-in-out. 호버/색 변화 → ease. 등속(마키·프로그레스) → linear.
어떻게기본 CSS easing은 약하다. 커스텀 커브를 쓴다. UI에 ease-in은 절대 금지 — 시작이 굼떠서, 사용자가 가장 집중해 보는 첫 순간을 잡아먹는다.
/* 강한 커스텀 커브 */
--ease-out:    cubic-bezier(0.23, 1, 0.32, 1);
--ease-in-out: cubic-bezier(0.77, 0, 0.175, 1);
--ease-drawer: cubic-bezier(0.32, 0.72, 0, 1); /* iOS 드로어 */
ease-in느리게 시작 → 굼떠 보임
ease-out (강한 커브)즉시 반응
linear기계적

커브는 직접 만들지 말고 easing.dev · easings.co에서 강한 변형을 고르세요.

④ 얼마나 빨라야 하나?

실제 예시
언제UI 모션은 300ms 미만. 180ms 드롭다운이 400ms보다 더 반응 좋게 느껴집니다.
어떻게체감 성능: 같은 로딩이라도 빠르게 도는 스피너가 더 빨라 보인다. 속도는 ease가 증폭한다 — 200ms ease-out이 200ms ease-in보다 빠르게 느껴짐.
요소권장 지속
버튼 누름 피드백100–160ms
툴팁, 작은 팝오버125–200ms
드롭다운, 셀렉트150–250ms
모달, 드로어200–500ms
마케팅/설명더 길어도 됨
0.6s · 빨라 보임
1.6s · 느려 보임
03

스프링 애니메이션

실제 물리를 흉내 내 더 자연스럽다. 고정 지속시간이 없고 파라미터로 정착합니다.

언제 스프링을 쓰나

실제 예시
언제모멘텀이 있는 드래그 · ‘살아있는’ 느낌(다이내믹 아일랜드) · 중간에 끊길 수 있는 제스처 · 장식적 마우스 트래킹.
어떻게Apple식 설정이 추론하기 쉽다. bounce는 0.1–0.3으로 작게, 대부분의 UI에선 bounce를 피하고 드래그-투-디스미스 같은 곳에만.
피하기은행 앱의 기능적 그래프처럼 의미가 있는 곳엔 차라리 무애니메이션이 낫다. 스프링은 장식일 때 빛난다.
// Apple식 (권장)
{ type: "spring", duration: 0.5, bounce: 0.2 }
// 전통 물리 (더 세밀)
{ type: "spring", mass: 1, stiffness: 100, damping: 10 }
↓

위 데모는 CSS 오버슈트로 흉내 낸 것 — 실제 스프링은 Motion의 useSpring으로 구현합니다.

끊겨도 속도를 유지한다

언제사용자가 도중에 마음을 바꿀 수 있는 제스처.
어떻게스프링은 중단 시 현재 속도를 유지하지만 CSS keyframe은 0에서 다시 시작한다. 펼친 항목을 클릭했다가 곧장 Esc를 누르면, 스프링은 현재 위치에서 부드럽게 되감긴다.

컴포넌트 제작 원칙

개별 컴포넌트를 만질 때 반복해서 쓰이는 핵심 규칙들.

버튼은 눌리는 느낌이 나야 한다

실제 예시
언제누를 수 있는 모든 요소.
어떻게:active에 scale(0.95–0.98). 즉각적인 피드백이 ‘인터페이스가 듣고 있다’는 느낌을 준다.
.button { transition: transform 160ms ease-out; }
.button:active { transform: scale(0.97); }
:active scale(0.96) 적용
손이 닿는 느낌

scale(0)에서 등장하지 마라

실제 예시
언제요소가 나타나는 모든 진입 애니메이션.
어떻게현실의 무엇도 무에서 튀어나오지 않는다. scale(0.9) 이상 + opacity로 시작 — 바람 빠진 풍선도 형체는 있듯이.

팝오버는 origin-aware하게

실제 예시
언제트리거에 붙는 팝오버·드롭다운·툴팁.
어떻게트리거에서 자라나야 한다. 기본 transform-origin: center는 거의 모든 팝오버에서 틀렸다. 단, 모달은 예외 — 특정 트리거에 묶이지 않으니 center 유지.
/* Radix */ transform-origin: var(--radix-popover-content-transform-origin);
/* Base UI */ transform-origin: var(--transform-origin);

툴팁: 두 번째부터는 지연 없이

실제 예시
언제툴바처럼 툴팁이 모여 있는 곳.
어떻게오작동 방지를 위해 첫 툴팁은 지연 후 등장. 하지만 하나가 열린 뒤 옆으로 옮기면 지연·애니메이션 없이 즉시 — 툴바 전체가 빨라 보인다.

하나에 올린 뒤 옆 버튼으로 빠르게 옮겨보세요 — 지연 없이 즉시 뜹니다.

동적 UI엔 keyframe보다 transition

실제 예시

어떻게transition은 도중에 가로채 새 목표로 재조준되지만 keyframe은 0에서 재시작한다. 아래 토스트를 연타해 보세요 — 부드럽게 스택됩니다.

블러로 어색한 전환을 가린다

실제 예시
언제크로스페이드가 easing·duration을 바꿔도 어색할 때.
어떻게전환 중 filter: blur(2–4px). 블러가 두 상태를 섞어, 겹친 두 객체가 아니라 하나의 변형으로 속인다. Safari에서 비싸니 20px 미만.

진입 애니메이션 (@starting-style)

실제 예시
언제JS 없이 요소 진입을 애니메이션하고 싶을 때.
어떻게React의 useEffect → mounted:true 패턴을 대체. 브라우저 지원이 되면 @starting-style, 아니면 data-mounted 폴백.
.toast {
  opacity: 1; transform: translateY(0);
  transition: opacity 400ms ease, transform 400ms ease;
  @starting-style { opacity: 0; transform: translateY(100%); }
}

CSS Transform 마스터리

transform을 제대로 쓰면 JS 없이도 멀리 간다.

translateY(%)는 자기 크기 기준

언제높이가 가변인 드로어/토스트를 화면 밖으로 숨길 때.
어떻게translateY(100%)는 실제 픽셀과 무관하게 ‘자기 높이만큼’ 이동. Sonner가 토스트를, Vaul이 드로어를 숨기는 방식. 하드코딩 px보다 안전.
.drawer-hidden { transform: translateY(100%); }
.toast-enter   { transform: translateY(-100%); }

scale()은 자식까지 같이 키운다

언제버튼을 누를 때 scale.
어떻게width/height와 달리 scale은 폰트·아이콘·내용까지 비례해 함께 키운다. 버그가 아니라 기능 — 누름 피드백이 통째로 자연스럽다.

06

clip-path 애니메이션

clip-path는 도형용이 아니라, CSS에서 가장 강력한 애니메이션 도구 중 하나입니다.

inset 기초

언제요소를 한쪽에서부터 드러내거나 가릴 때.
어떻게inset(top right bottom left) — 각 값이 그 변에서 안쪽으로 ‘파먹는다’. transition으로 부드럽게 reveal.
.hidden  { clip-path: inset(0 100% 0 0); } /* 오른쪽에서 완전히 가림 */
.visible { clip-path: inset(0 0 0 0);   } /* 전부 보임 */

탭의 완벽한 색 전환

언제활성 탭과 비활성 탭의 글자색이 다를 때.
어떻게탭 목록을 복제해 사본을 ‘활성’ 스타일로. 활성 탭만 보이도록 clip하고 탭 변경 시 clip을 애니메이션. 개별 색 전환 타이밍으론 못 내는 매끈함.

꾹 눌러 삭제 (hold-to-delete)

실제 예시
언제실수 방지가 필요한 파괴적 동작.
어떻게colored overlay를 inset(0 100% 0 0)로 두고 :active에 2s linear로 inset(0 0 0 0). 놓으면 200ms ease-out로 스냅백 — 비대칭 타이밍의 정석.
누름: 1.6s linear · 놓음: 200ms ease-out (비대칭)

비교 슬라이더

실제 예시
언제Before/After를 겹쳐 비교할 때.
어떻게두 이미지를 겹치고 위 이미지를 clip-path: inset(0 50% 0 0). 드래그 위치로 right inset만 조절 — 추가 DOM 없이 GPU 가속.
After · ease-out 180ms
Before · ease-in 450ms

스크롤 진입 이미지 리빌

언제뷰포트에 들어올 때 이미지를 드러낼 때.
어떻게inset(0 0 100% 0)(아래에서 가림)으로 시작 → 보이면 inset(0 0 0 0). IntersectionObserver나 Framer Motion useInView({ once:true, margin:'-100px' }).
07

제스처 & 드래그 인터랙션

현실의 물체는 갑자기 멈추지 않습니다. 제스처도 그래야 자연스럽습니다.

모멘텀 기반 닫기

실제 예시
언제스와이프로 닫는 토스트·카드·드로어.
어떻게임계 거리만 보지 말고 속도를 계산: |거리| / 경과시간. 속도가 ~0.11을 넘으면 거리와 무관하게 닫는다. 빠른 플릭 한 번이면 충분.
const velocity = Math.abs(swipeAmount) / timeTaken;
if (Math.abs(swipeAmount) >= THRESHOLD || velocity > 0.11) dismiss();
🔔
새 알림
좌우로 휙 밀어 닫기 (빠르게 튕기면 짧게 밀어도 닫힘)
속도 > 0.11px/ms 또는 120px 이상 → 닫힘

경계에서 댐핑 / 하드 스톱 대신 마찰

언제자연 경계를 넘겨 드래그할 때(이미 맨 위인 드로어를 더 끌 때).
어떻게많이 끌수록 덜 움직이게 감쇠. 위로 못 끌게 막는 대신 마찰을 키워 허용 — 보이지 않는 벽에 부딪히는 것보다 자연스럽다. 위 카드를 끝까지 밀어보면 저항이 붙습니다.

포인터 캡처 / 멀티터치 보호

언제드래그가 요소 밖으로 나가거나, 도중에 손가락이 바뀔 때.
어떻게드래그 시작 시 setPointerCapture로 모든 포인터 이벤트를 잡아 밖으로 나가도 이어지게. 첫 드래그 후 추가 터치는 무시 — 손가락 교체 시 점프 방지.
function onPress(e) {
  if (isDragging) return;       // 멀티터치 무시
  el.setPointerCapture(e.pointerId);
}
08

성능 규칙

부드러움은 GPU 위에서 나옵니다. 잘못된 속성 하나가 프레임을 떨어뜨립니다.

transform과 opacity만 애니메이션하라

언제60fps가 필요한 모든 곳(사실상 전부).
어떻게이 둘은 layout·paint를 건너뛰고 GPU에서 돈다. padding·margin·height·width를 애니메이션하면 세 단계 모두 유발.

CSS 변수는 상속된다 — 자식에 비싸다

언제드로어처럼 자식이 많은 컨테이너를 드래그할 때.
어떻게부모의 CSS 변수를 바꾸면 모든 자식의 스타일이 재계산된다. --swipe-amount 대신 해당 요소의 transform을 직접 갱신.
// 나쁨: 모든 자식 재계산
el.style.setProperty('--swipe-amount', d + 'px');
// 좋음: 이 요소만
el.style.transform = 'translateY(' + d + 'px)';

Framer Motion x/y는 GPU 가속이 아니다

언제페이지 로딩 등 메인 스레드가 바쁜 와중에 애니메이션할 때.
어떻게x/y/scale 단축 속성은 rAF로 메인 스레드에서 돈다. 하드웨어 가속하려면 transform 전체 문자열을 써라. Vercel 대시보드 탭이 로딩 중 프레임을 떨어뜨린 걸 CSS로 바꿔 해결.
<motion.div animate={{ transform: "translateX(100px)" }} /> // 가속됨

정해진 모션은 CSS, 동적 모션은 WAAPI

언제미리 정해진 애니메이션 vs 프로그램적 제어가 필요할 때.
어떻게CSS 애니메이션은 메인 스레드 밖에서 돌아 로딩 중에도 매끈. 프로그램 제어가 필요하면 Web Animations API — JS 제어 + CSS 성능, 라이브러리 불필요.
el.animate(
  [{ clipPath: 'inset(0 0 100% 0)' }, { clipPath: 'inset(0 0 0 0)' }],
  { duration: 1000, fill: 'forwards', easing: 'cubic-bezier(0.77,0,0.175,1)' }
);
09

접근성

모션은 멀미를 유발할 수 있습니다. 줄이되, 0으로 만들지는 마세요.

prefers-reduced-motion

실제 예시
언제모든 모션. 사용자가 OS에서 동작 줄이기를 켰을 때.
어떻게더 적고 부드럽게 — 이동·위치 애니메이션은 제거하되, 이해를 돕는 opacity·색 전환은 유지.
@media (prefers-reduced-motion: reduce) {
  .element { animation: fade .2s ease; /* transform 이동 없음 */ }
}
확인 중… OS 설정(손쉬운 사용 → 동작 줄이기)을 바꾸면 즉시 반영됩니다.

터치 기기의 호버 상태

언제호버에 모션을 거는 모든 곳.
어떻게터치 기기는 탭에도 호버가 켜져 오작동한다. (hover: hover) and (pointer: fine) 뒤에 호버 애니메이션을 게이트.
@media (hover: hover) and (pointer: fine) {
  .element:hover { transform: scale(1.05); }
}
10

Sonner 원칙 — 사랑받는 컴포넌트 만들기

주간 1,300만 다운로드의 Sonner를 만들며 얻은, 어떤 컴포넌트에도 통하는 원칙.

여섯 가지 원칙

  1. 개발자 경험이 핵심. 훅·컨텍스트·복잡한 설정 없이 한 번 설치, 어디서든 호출. 마찰이 적을수록 많이 쓴다.
  2. 옵션보다 좋은 기본값. 대부분 커스텀하지 않는다. 기본 easing·타이밍·디자인이 훌륭해야 한다.
  3. 이름이 정체성을 만든다. “Sonner”(프랑스어 ‘울리다’)가 “react-toast”보다 우아하다. 때론 검색성보다 기억성.
  4. 엣지 케이스를 보이지 않게. 탭이 숨겨지면 타이머 일시정지, 스택 사이 틈을 의사요소로 메워 호버 유지, 드래그 중 포인터 캡처.
  5. 동적 UI엔 keyframe 아닌 transition. 토스트는 빠르게 추가되고, keyframe은 중단 시 0에서 재시작한다.
  6. 훌륭한 문서 사이트. 직접 만져보고 이해한 뒤 쓰게. 바로 복사할 인터랙티브 예시가 채택 장벽을 낮춘다.

응집(cohesion)이 만족감을 만든다

언제컴포넌트의 ‘성격’을 정할 때.
어떻게Sonner는 일부러 살짝 느리고 ease-out 대신 ease로 더 우아하게 — 모션·디자인·이름이 한 결로 어울린다. 장난스러운 컴포넌트는 더 통통 튀게, 전문적 대시보드는 또렷하고 빠르게. 모션을 무드에 맞춰라.

비대칭 enter/exit 타이밍

언제사용자가 ‘결정’하는 구간과 시스템이 ‘응답’하는 구간이 나뉠 때.
어떻게결정이 신중해야 하면 느리게(꾹 눌러 삭제: 2s linear), 응답은 늘 빠르게(200ms ease-out). 위 hold-to-delete 데모가 바로 이 패턴입니다.

opacity + height 조합 / 다음 날 다시 보기

언제목록에 항목이 들고 날 때 / 애니메이션을 완성했다고 느낄 때.
어떻게opacity 변화와 height 애니메이션의 궁합은 공식이 없다 — 맞을 때까지 조정. 그리고 다음 날 맑은 눈으로 다시 보면 어제 못 본 흠이 보인다. 슬로모·프레임 단위로 재생해 점검.

스태거 애니메이션

여럿이 함께 들어올 땐, 약간씩 시차를 두면 한꺼번에 나타나는 것보다 자연스럽습니다.

짧은 시차로 캐스케이드

실제 예시
언제목록·그리드·메뉴가 한 번에 등장할 때.
어떻게항목마다 30–80ms 시차. 너무 길면 느려 보인다. 스태거는 장식 — 재생 중 인터랙션을 막지 마라.
.item:nth-child(2) { animation-delay: 50ms; }
.item:nth-child(3) { animation-delay: 100ms; }

애니메이션 디버깅

전속력에선 안 보이는 흠이 느린 화면에선 드러납니다. 위의 ‘슬로모 4×’ 토글이 바로 이 도구예요.

슬로모 테스트

언제easing·타이밍·origin이 맞는지 의심될 때.
어떻게지속시간을 2–5배로 늘리거나 DevTools 애니메이션 패널로 느리게 재생. 색이 두 상태로 겹쳐 보이는지, easing이 뚝뚝 끊기는지, origin이 엉뚱한 점인지, 여러 속성이 동기화되는지 확인.

프레임 단위 검사 / 실기기 테스트

언제좌표된 속성 간 타이밍 / 터치 제스처 검증.
어떻게Chrome DevTools Animations 패널로 프레임을 한 칸씩. 드로어·스와이프는 반드시 실제 기기에서 — USB로 폰을 연결해 IP로 접속하고 Safari 원격 devtools. 시뮬레이터보다 실물이 정확.
